# Skill Review - Initial Assessment Phase

This document provides detailed guidance for Steps 1-3 of the skill-review workflow.

## Step 1: Load Skill/Workflow with Fresh Context

### Fresh Context Approach

The key to effective review is to approach the skill/workflow as if encountering it for the first time, without prior assumptions or context about its creation.

**Why fresh context matters**:
- Removes bias from creation process
- Identifies gaps that creators overlook
- Simulates real user experience
- Uncovers implicit assumptions

### Loading Process

**Read the main file**:
- Read SKILL.md or workflow markdown file completely (read the entire file, paying attention to structure, content, and organization)
- Note the structure and organization
- Identify the overall flow
- Mark sections that are unclear

**Read supporting files**:
- Read all referenced supporting files
- Understand how they relate to the main file
- Note any missing or incomplete references
- Check if file references are accurate

**Check file structure**:
- Verify directory structure exists
- Check all referenced files exist
- Verify file permissions are correct
- Note any structural issues

### Context Loading Checklist

- [ ] Main skill/workflow file read completely
- [ ] All supporting files located and read
- [ ] File structure verified
- [ ] References checked for accuracy
- [ ] Initial notes taken on structure
- [ ] Questions documented for later investigation

### Common Issues to Identify

**Missing files**:
- Referenced files don't exist
- Broken file paths
- Incorrect file names

**Incomplete references**:
- Files mentioned but not referenced
- Vague references to "see documentation"
- No clear indication of when to load files

**Structural issues**:
- Poor organization
- Inconsistent formatting
- Missing sections
- Unclear hierarchy

## Step 2: Understand Intent and Goals

### Intent Analysis

Analyze what the skill/workflow is intended to accomplish:

**Stated purpose**:
- What does the description say it does?
- What is the stated goal?
- Who is the intended user?
- When should it be used?

**Implied purpose**:
- What does the content suggest it does?
- What patterns emerge from the steps?
- What use cases are implied?
- What problems does it solve?

### Goal Validation

**Primary goals**:
- Identify the main objective
- Verify it's achievable
- Check if it's clearly stated
- Assess if it's appropriate

**Secondary goals**:
- Identify additional objectives
- Check for goal conflicts
- Assess priority of goals
- Verify alignment with primary goal

### Intent Clarity Assessment

**Clear intent indicators**:
- Specific, measurable goals
- Clear use case descriptions
- Well-defined scope
- Explicit success criteria

**Unclear intent indicators**:
- Vague descriptions
- Multiple conflicting goals
- Undefined scope
- Missing success criteria

### Questions to Ask

- What problem is this solving?
- Who will use this?
- When will it be used?
- What should the outcome be?
- Is this the right approach?

### Intent Documentation

Document your understanding:
```
Intent: [What the skill/workflow intends to do]
Primary Goal: [Main objective]
Secondary Goals: [Additional objectives]
Target User: [Who will use it]
Use Cases: [When it will be used]
Success Criteria: [What defines success]
```

## Step 3: Identify Scope and Boundaries

### Scope Definition

**What's included**:
- Specific tasks covered
- Technologies addressed
- File types supported
- Environments applicable

**What's excluded**:
- Tasks explicitly out of scope
- Technologies not covered
- File types not supported
- Environments not applicable

### Boundary Analysis

**Explicit boundaries**:
- Stated inclusions/exclusions
- Documented limitations
- Specified prerequisites
- Defined constraints

**Implicit boundaries**:
- Implied by content
- Suggested by examples
- Indicated by structure
- Evident from steps

### Scope Clarity Assessment

**Clear scope indicators**:
- Explicit inclusions/exclusions
- Well-defined boundaries
- Clear prerequisites
- Stated limitations

**Unclear scope indicators**:
- Vague boundaries
- Missing prerequisites
- Undefined limitations
- Ambiguous inclusions

### Scope Issues to Identify

**Scope creep**:
- Trying to do too much
- Unrelated features included
- Loss of focus
- Overly broad scope

**Scope gaps**:
- Missing necessary functionality
- Incomplete coverage
- Unaddressed use cases
- Missing edge cases

**Scope conflicts**:
- Conflicting goals
- Incompatible features
- Contradictory requirements
- Mutually exclusive capabilities

### Boundary Validation

**Prerequisite boundaries**:
- Are prerequisites realistic?
- Are they necessary?
- Are they sufficient?
- Are they clearly stated?

**Technology boundaries**:
- Are technology choices appropriate?
- Are they current?
- Are they supported?
- Are they documented?

**Environment boundaries**:
- Are environment assumptions valid?
- Are they documented?
- Are they testable?
- Are they reasonable?

### Scope Documentation

Document scope findings:
```
Included: [What's in scope]
Excluded: [What's out of scope]
Prerequisites: [Required conditions]
Assumptions: [What's assumed]
Limitations: [Known limitations]
Scope Issues: [Any scope problems identified]
```
